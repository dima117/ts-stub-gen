# stub-gen — дизайн

## Идея

Для каждого типа данных генерируется функция-хелпер, возвращающая объект,
заполненный значениями по умолчанию. Хелпер принимает параметр
`overrides: Partial<T>` — в тесте передаются только поля, которые важны
для теста, остальные заполняются дефолтами. Хелперы компонуются: для
вложенного поля в overrides передаётся результат хелпера вложенного типа.

## Архитектура: два этапа

```
источник (типы TS | swagger | GraphQL) → [парсер] → схема → [генератор] → API (TS | ...)
```

- **Схема** — декларативное промежуточное представление, сериализуемое в JSON
  (никаких функций и ссылок на компилятор). Схема — контракт между этапами и
  предмет проверки в тестах парсера.
- Приоритетная пара: источник — типы TypeScript, назначение — API на TypeScript.

## Схема

Типы схемы: [src/schema.ts](../src/schema.ts).

Ключевые решения:

- **Идентификатор типа = `(namespace, name)`.** Для TypeScript namespace —
  путь файла без расширения (`models/user`). Схема трактует namespace как
  непрозрачную строку; это решает коллизии одинаковых имён в разных файлах.
- **Схема монофорфна.** Открытые дженерики (`Paginated<T>`) в схему не
  попадают; инстанцирования (`Paginated<Account>`) раскрываются структурно по
  месту использования. Утилитарные типы (`Partial`, `Pick`, `Omit`,
  `Required`, `Readonly`), пересечения (`A & B`), mapped types и indexed
  access раскрываются компилятором на этапе парсинга.
- **Именованное — ссылкой (`ref`), анонимное — инлайн.** Любое именованное
  объявление (interface, type alias, enum) становится определением;
  использование — узлом `ref`. Неэкспортированные типы попадают в схему с
  `exported: false` — решение разворачивать их инлайн принимает генератор.
  `ref` указывает на место объявления: переименованные импорты и реэкспорты
  разрезолвлены.
- **`a?: string` ≠ `a: string | undefined`.** Первое даёт `optional: true` +
  `string` (неявный undefined убирается), второе — `optional: false` +
  `union(string, undefined)`.
- **Порядок вариантов union — как в исходнике** (генератор берёт первый),
  поэтому парсер читает синтаксис (TypeNode), а не resolved-тип. `boolean`,
  расщеплённый компилятором на `true | false` внутри вычисляемых типов,
  склеивается обратно.
- **enum остаётся именованным определением** (не union литералов): генератору
  TS нужно эмитить `EnumName.Member` и импорт.
- **`extends`** — поля базовых интерфейсов вливаются в объект; собственное
  поле переопределяет одноимённое базовое, сохраняя его позицию.
- **`Date` — отдельный `kind: "date"`.** Date в данных нарушает
  JSON-сериализуемость, поэтому есть настраиваемое предупреждение
  `date-type` (по умолчанию выключено).
- **Определения отсортированы** по `(namespace, name)` — вывод детерминирован.

## Предупреждения

Диагностика — не часть схемы: парсер возвращает `{ schema, warnings }`.
У каждого предупреждения есть fallback-поведение (применяется всегда) и
настраиваемый уровень `off | warn | error`
(`ParserOptions.warningLevels`). Коды и уровни по умолчанию:

| Код | Когда | Fallback | По умолчанию |
| --- | --- | --- | --- |
| `behavior-in-type` | функция/метод/сигнатура вызова в типе | `unknown` | `warn` |
| `unsupported-type` | классы, внешние типы (Map, Set, Promise…), прочее | `unknown` | `warn` |
| `date-type` | поле типа `Date` | `kind: "date"` | `off` |
| `default-export` | `export default` объявления | пропуск / `unknown` | `warn` |
| `expansion-depth` | превышена глубина раскрытия (рекурсивный дженерик) | `unknown` | `warn` |

Уровень `error` пока только помечает предупреждение; CLI будет завершаться
с ненулевым кодом (позже).

## Тесты парсера

Кейс = виртуальная файловая структура (путь → содержимое) + ожидаемая схема +
ожидаемые коды предупреждений. Используется in-memory файловая система
ts-morph — импорты между виртуальными файлами резолвятся честно.

Формат: [tests/types.ts](../tests/types.ts), билдеры выражений:
[tests/dsl.ts](../tests/dsl.ts), раннер: [tests/schema.test.ts](../tests/schema.test.ts).

Группы кейсов ([tests/cases](../tests/cases)):

- **A. Примитивы и литералы** — string/number/boolean, литеральные типы,
  null/undefined, any/unknown, порядок union, optional vs `| undefined`.
- **B. Структуры** — вложенные анонимные объекты, все синтаксисы массивов,
  кортежи, индексные сигнатуры и `Record`.
- **C. Именованные типы и ссылки** — ссылки на интерфейсы, неэкспортированные
  типы, алиасы (примитив/union/объект), enum, рекурсия, extends.
- **D. Файлы и импорты** — импорт из другого файла, коллизии имён,
  `import type`, переименование, реэкспорты.
- **E. Вычисляемые типы** — инстанцирование дженериков, Partial/Pick/Omit,
  пересечения, mapped types, indexed access.
- **F. Диагностика** — функции в типах, классы и контейнеры, Date,
  default export.

## Генератор TS (этап 2)

Реализация: [src/generator](../src/generator). Вход — схема, выход — текст
одного TS-файла ([generate.ts](../src/generator/generate.ts)).

- **Хелпер для каждого определения с `exported: true`**, включая алиасы
  примитивов и enum: для алиасов потом появятся кастомные значения по
  умолчанию (например, алиас PhoneNumber → валидный номер вместо `""`).
- **Сигнатура зависит от тела определения:**
  - `object` → `(overrides: Partial<T> = {}) => ({ ...defaults, ...overrides })` —
    плоский merge, вложенный объект заменяется целиком (композиция хелперов);
  - `record` → `(overrides: T = {}) => ({ ...overrides })` — `Partial` индексной
    сигнатуры добавил бы `undefined` в тип значений;
  - остальное (примитивы, union, tuple, enum, date) → `(override?: T)` —
    подмена значения целиком.
- **Optional-поля не заполняются**: стаб минимален, и это разрывает рекурсию
  хелперов при взаимных ссылках через optional-поля. Политика — кандидат на
  настройку (TODO).
- Значения по умолчанию по kind: `string` → `""`, `number` → `0`,
  `boolean` → `false`, `literal` → значение, `union` → первый вариант,
  `array` → `[]`, `tuple` → по элементам, `record` → `{}`,
  `object` → рекурсивно, `date` → `new Date(0)`, `null` → `null`,
  `undefined` → `undefined`, `unknown` → `undefined as any`.
- **`ref`**: на экспортированный тип — вызов его хелпера
  (`account: GetStubAccount()`); на неэкспортированный — инлайн-раскрытие тела
  (цикл → `undefined as any` + предупреждение `inline-cycle`). Первый член
  неэкспортированного enum подставляется значением с `as any`.
- **Имена**: `GetStub<Имя>`; при экспорте одинаковых имён из разных namespace —
  суффикс из namespace (`GetStubItem_a_item`) и импорт типа с алиасом
  (`import type { Item as Item_a_item }`).
- **Импорты**: `import type` для интерфейсов/алиасов, обычный `import` для
  enum (нужны в рантайме); спецификаторы — относительные пути от
  `outputNamespace` выходного файла.
- **Предупреждения генератора** — тот же механизм (fallback + уровень
  `off | warn | error`, общий [src/warnings.ts](../src/warnings.ts)):
  `ref-not-found`, `inline-cycle`.
- Известное ограничение: обязательное рекурсивное поле
  (`next: LinkedList`) даёт бесконечную рекурсию хелперов в рантайме (TODO).

Тесты генератора:
- [tests/generator.test.ts](../tests/generator.test.ts) — точный текст
  сгенерированного кода для представительных схем + рантайм-проверка дефолтов
  и merge overrides (генерация → transpile → выполнение);
- [tests/generator-compile.test.ts](../tests/generator-compile.test.ts) —
  свойство: код, сгенерированный по схеме каждого кейса парсера, обязан
  компилироваться без ошибок рядом с исходными файлами кейса.

## CLI

Реализация: [src/cli](../src/cli) — `bin.ts` (исполняемый вход),
`index.ts` (аргументы и режимы), `config.ts` (загрузка и валидация конфига),
`run.ts` (конвейер парсер → генератор → запись).

- Запуск: `stub-gen [путь-к-конфигу]`; по умолчанию — `./stub-gen.config.json`
  в текущей папке. Режимы `--help`/`-h` и `--version`/`-v`.
- Конфиг — JSON, все пути относительно файла конфига:
  - `source.type` — тип источника (пока только `"typescript"`);
  - `source.entry` — файл/глоб или их массив;
  - `source.rootDir` — корень для namespace типов (по умолчанию папка конфига);
  - `source.tsconfig` — необязательный tsconfig для настроек компилятора;
  - `output.file` — путь генерируемого файла (namespace выходного файла
    считается от rootDir — от него строятся относительные импорты);
  - `output.helperPrefix` — префикс имён хелперов (по умолчанию `GetStub`);
  - `warnings` — уровни `off | warn | error` по кодам парсера и генератора
    (один словарь, CLI сам раскладывает по этапам).
- Конфиг описан **схемой valibot** ([config.ts](../src/cli/config.ts)) —
  единственный источник правды: тип `CliConfig` выводится из схемы
  (`v.InferOutput`), валидация (включая неизвестные поля через `strictObject`
  и коды предупреждений через `picklist`) выполняется по ней же. valibot
  бандлится в dist и в рантайм-зависимости пакета не попадает.
- Аргументы CLI описываются декларативно через **cac** (минималистичный
  парсер, на нём работает CLI vite): флаги, команда и версия объявляются один
  раз, **справка генерируется** из объявлений — рукописного help-текста нет.
  Неизвестные флаги и лишние аргументы отклоняет сам cac. Структура конфига
  в справку не дублируется — справка отсылает к документации пакета.
- **Никаких захардкоженных идентификаторов в сообщениях**: имя утилиты
  берётся из `bin` в package.json ([constants.ts](../src/cli/constants.ts)),
  конфиг по соглашению — производное `<утилита>.config.json`, префикс
  хелперов/уровни предупреждений/типы источников — константы. Путь артефакта
  сборки vite тоже выводится из `bin`. Подсказку «справка: …» к ошибкам
  добавляет только `main()` — ошибки конфига не знают о способе запуска CLI.
- cac, как и valibot, бандлится в dist — рантайм-зависимость пакета
  по-прежнему одна (`ts-morph`).
- Для всех необязательных настроек есть fallback внутри CLI; ошибки валидации
  включают dot-path до проблемного поля.
- **Предупреждения уровня `error` прерывают генерацию**: файл не записывается,
  код выхода 1.
- Сборка: vite, lib-режим (cjs, node18), `ts-morph` и встроенные модули —
  external; результат `dist/stub-gen.cjs` подключён через `bin` пакета.
- Тесты: [tests/cli.test.ts](../tests/cli.test.ts) — разбор аргументов,
  валидация конфига, интеграционные прогоны во временной папке на реальной ФС.
